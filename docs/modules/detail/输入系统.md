# 输入系统详细分析

## 1. 模块职责

输入系统负责把用户的键盘和鼠标操作，转换成游戏可以理解的语义动作。

主要类：

- `KeyboardManager`
- `MouseManager`
- `InputAction`
- `InputActionMapper`
- `GameInputController`
- `KeyBindingsWindow`

## 2. 输入层分工

### 2.1 `KeyboardManager`

它只做两件事：

1. 记录当前按下的键
2. 记录“本帧刚按下”的键

这种设计非常适合游戏循环，因为很多动作只需要触发一次，而不是持续触发。

### 2.2 `MouseManager`

鼠标管理器保存：

- 当前按下的按钮
- 本帧刚点击的按钮
- 鼠标坐标

因此同一套输入逻辑既可以用于菜单，也可以用于战斗和编辑器。

### 2.3 `InputAction`

动作枚举不是“按键枚举”，而是“语义枚举”。

例如：

- 移动
- 跳跃
- 射击
- 暂停
- 切换弹种
- 切换油门
- 建造 / 销毁体素
- 关卡目标开关

这样做的好处是：后续可以随时换按键，但游戏逻辑不用变。

## 3. 默认映射

`InputActionMapper.createDefaultGameMapping()` 里定义了默认输入布局。

| 动作 | 默认输入 |
| --- | --- |
| 向上移动 | `W` |
| 向下移动 | `S` |
| 向左移动 | `A` |
| 向右移动 | `D` |
| 油门上 | `↑` |
| 油门下 | `↓` |
| 油门左 | `←` |
| 油门右 | `→` |
| 上一项菜单 | `Q` / `W` / `↑` |
| 下一项菜单 | `E` / `S` / `↓` |
| 确认菜单 | `Enter` / `Space` / 鼠标左键 |
| 暂停 | `Esc` / `P` |
| 对话继续 | `Space` / `Enter` |
| 切换颜色 | `C` |
| 切换弹种 | `B` |
| 体素建造 | 鼠标中键 |
| 体素销毁 | 鼠标右键 |
| 跳跃 | `Space` |
| 射击 | `K` / 鼠标左键 |
| 冲刺 | `Shift` |
| 切换油门 | `T` |
| 显示目标 | `G` |

> 说明：键位可通过后续的按键绑定界面和设置持久化重新配置。

## 4. 处理流程

### 4.1 `GameInputController.processInputs()`

该方法首先判断当前世界是否绑定了状态机：

- **有状态机**：把输入交给状态机处理
- **没有状态机**：直接处理玩家移动、菜单导航和体素系统

这保证了“状态驱动”和“自由处理”两种场景都能工作。

### 4.2 玩家移动

玩家移动处理逻辑包含以下步骤：

1. 检查 UI 是否正在覆盖游戏画面
2. 响应颜色切换和弹种切换
3. 处理跳跃
4. 处理射击或炸弹蓄力
5. 处理油门切换
6. 计算移动向量
7. 处理冲刺与体力消耗
8. 体力不足或未移动时恢复体力

这里的设计重点是：**输入不是简单的“按键到位移”，而是和角色状态绑定的行为系统。**

### 4.3 射击与炸弹

射击逻辑区分两类：

- **普通射击**：按下即可开火
- **炸弹类型**：按住蓄力，松开发射

蓄力时还会记录方向：

- 键盘方向
- 鼠标方向

因此玩家既可以面向最后移动方向射击，也可以直接朝鼠标位置射击。

### 4.4 菜单导航

菜单导航支持：

- 上一项 / 下一项
- 鼠标悬停
- 鼠标点击确认
- 键盘确认

菜单选择后还会同步到对话框，方便调试和提示当前项。

### 4.5 体素系统

体素系统支持：

- 鼠标中键建造体素
- 鼠标右键销毁体素
- 网格对齐

这让游戏本体和编辑器在交互上保持一致。

## 5. 每帧收尾

`finishFrame()` 会清理 transient 状态：

- 本帧按下
- 本帧点击

这样下一帧就不会重复触发一次性动作。

## 6. 相关测试

输入系统相关测试主要关注：

- 默认映射是否识别键盘和鼠标动作
- 清除键位绑定是否不影响鼠标绑定
- WASD 移动是否驱动玩家移动
- 冲刺是否消耗体力
- 停止移动后是否恢复体力
- 鼠标/键盘菜单选择是否正常
- 左键射击、炸弹蓄力、切换弹种、中键建造是否生效
