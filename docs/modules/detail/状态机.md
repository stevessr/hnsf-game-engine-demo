# 状态机详细分析

## 1. 模块职责

状态机是整个游戏“行为切换”的核心。

它负责：

- 判断当前状态
- 决定允许哪些输入
- 控制菜单与对话框显示
- 管理暂停、失败、胜利、结算和重开关卡
- 通过运行时动作把请求传递给应用层

主要类：

- `GameState`
- `GameStateMachine`
- `GameStateContext`
- `GameRuntimeActions`
- `GameSettings`
- `DefaultGameStateMachine`

## 2. 状态分类

`GameState` 目前包含：

- `MENU`
- `PLAYING`
- `DIALOG`
- `PAUSED`
- `GAMEOVER`
- `SETTLEMENT`

每个状态都带有自己的行为约束：

- `allowsPlayerMovement()`
- `allowsWorldUpdate()`
- `allowsMenuNavigation()`
- `allowsDialogInteraction()`

## 3. 状态转换关系

`DefaultGameStateMachine` 对允许的转换做了显式限制。

### 3.1 菜单状态

`MENU` 可以进入：

- `DIALOG`
- `PLAYING`

### 3.2 游戏状态

`PLAYING` 可以进入：

- `PAUSED`
- `DIALOG`
- `MENU`
- `GAMEOVER`
- `SETTLEMENT`

### 3.3 对话状态

`DIALOG` 可以进入：

- `PLAYING`
- `SETTLEMENT`

### 3.4 暂停、失败、结算

- `PAUSED` → `PLAYING` / `MENU`
- `GAMEOVER` → `PLAYING` / `MENU`
- `SETTLEMENT` → `PLAYING` / `MENU`

如果转换不合法，状态机会直接抛出异常。  
这是一种“尽早失败”的设计，便于发现逻辑错误。

## 4. 上下文与运行时动作

### 4.1 `GameStateContext`

状态机收到的是上下文，而不是只有世界对象。

上下文包含：

- `GameWorld`
- `GameInputController`
- `GameSettings`
- `GameRuntimeActions`
- `deltaSeconds`

这让状态机既能改状态，也能请求更高层级的应用动作。

### 4.2 `GameRuntimeActions`

运行时动作把“状态机之外的事情”抽象出来，包括：

- 退出程序
- 加载指定关卡
- 加载下一关
- 打开编辑器
- 请求生成程序化关卡

这样状态机不需要知道 `App` 里的具体实现细节。

## 5. 菜单与界面对象

状态机里有几个固定菜单名称：

- `main-menu`
- `level-select-menu`
- `pause-menu`
- `options-menu`
- `gameover-menu`
- `victory-menu`

它们分别用于：

- 主菜单
- 关卡选择
- 暂停
- 选项
- 失败
- 胜利

## 6. 暂停逻辑

`togglePause()` 会做三件事：

1. 如果正在暂停，则恢复游戏
2. 如果正在游戏，则进入暂停
3. 在暂停和恢复时处理对话框隐藏/恢复

暂停时还会清空玩家移动输入，避免恢复后角色继续漂移。

## 7. 选项菜单

状态机支持大量动态选项切换，例如：

- 分辨率
- FPS
- 油门力度
- 减速度
- 重力开关与重力强度
- 光照开关
- 环境光
- 光照倍率
- 音效总开关
- 总音量
- 受击、射击、菜单、效果音量
- UI 字体大小

这说明状态机不仅负责“游戏流程”，也承担了“运行设置入口”的职责。

## 8. 游戏结束与结算

### 8.1 `GAMEOVER`

失败时会显示失败原因，并给出：

- 重生
- 返回菜单

### 8.2 `SETTLEMENT`

胜利或关卡完成后会显示：

- 下一关（若存在）
- 返回菜单

## 9. 相关测试

状态机测试覆盖了很多关键路径：

- 主菜单开始游戏、退出、生成森林/洞穴
- 游戏中暂停/恢复
- 关卡重载
- 非法状态转换
- 关卡选择
- 结算进入下一关
- 已关闭对话框不会立刻重新弹出
- `recenterUI` 在不同视口/摄像机状态下的行为
- 游戏结束菜单的失败原因展示
- 失败后的重生流程
